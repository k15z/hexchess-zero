The training pipeline is designed to run as a distributed system. All coordination happens through S3, so components can run on any mix of hardware.

For experimentation on a single machine, `make docker-up` brings up workers, trainer, the evaluation service, and the dashboard against a local or remote S3 bucket — see the [Docker](#docker) section. The Kubernetes setup below is what's used for actual long-running training; it scales the same components across a Hetzner k3s cluster plus a separate GPU node for the trainer.

| Component | Infrastructure | Resources |
|-----------|---------------|-----------|
| Self-play workers | Hetzner k3s cluster | CPU pods (7 CPU / 14Gi memory each, scaled via replicas) |
| Trainer | GPU-capable node (separate k3s cluster) | GPU + CPU |
| Evaluation service | Hetzner k3s cluster | 2 replicas × (4 CPU / 12Gi memory). Each replica plays one paired mini-match at a time; throughput scales with replica count. |
| Dashboard | Hetzner k3s cluster | Lightweight CPU pod + cloudflared tunnel sidecar |
| Shared state | S3 (DigitalOcean Spaces / Cloudflare R2) | Object storage |

## Docker

All components are containerized with a multi-arch Docker image (amd64 + arm64):

```bash
# Start everything locally
make docker-up

# Stop
make docker-down
```

The `docker-compose.yml` runs workers, trainer, the evaluation service, and the dashboard.

## Kubernetes

Kubernetes manifests live in `k8s/`. The cluster runs on Hetzner Cloud servers using k3s.

### Scaling workers

Workers are stateless — just add more replicas or nodes. Each worker:

- Polls S3 for the latest model
- Runs MCTS self-play (CPU-bound, ~14Gi memory per pod)
- Uploads `.npz` batches to S3
- Sends heartbeats so the dashboard can track worker health

### Resource requirements

Worker memory usage is dominated by the MCTS search tree (arena-allocated nodes) and the inference model loaded in memory. The evaluation service keeps only the active candidate/incumbent ONNX sessions in process and spends the rest of its footprint on paired matches, which is why the deployment is sized at 12Gi per replica.

## Monitoring

### Dashboard

```bash
make dashboard
```

The training dashboard (`training/dashboard.py` + `dashboard.html`) is an HTTP server backed by an in-memory store (`training/dashboard_store.py`) that incrementally syncs from S3 in a background thread — using ETag caches on model metadata and version-scoped evaluation summaries plus a LIST-based diff on `state/evals/v{N}/games/` so HTTP requests never block on S3. It shows:

- Worker heartbeats and health
- Training progress (positions generated, current model version, loss trends)
- Gate and benchmark evidence for the current candidate
- Recent immutable evaluation games for the focused version

### Status command

```bash
make status
```

Shows a quick summary of the training pipeline: active workers, approved and latest model pointers, position counts, and current evaluation status.

## Configuration

All configuration lives in `training/config.py` as a dataclass. S3 credentials are in `.env` (gitignored):

```
BUCKET_NAME=your-bucket
ACCESS_KEY=your-key
SECRET_KEY=your-secret
ENDPOINT=https://your-s3-endpoint
```

S3 key constants are defined in `training/storage.py` — never use raw string keys elsewhere.
