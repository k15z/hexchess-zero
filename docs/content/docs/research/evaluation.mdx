The evaluation path is intentionally narrow. It does **not** maintain a standing rating league, a cross-version ladder, or a global `elo.json` projection.

Its job is only to answer one question for each newly exported candidate:

- does this candidate beat the currently approved model?
- does it avoid regressing against the fixed anchor suite?
- can we explain the decision with immutable, version-scoped artifacts?

![Evaluation service dashboard showing the current baseline, gate, benchmarks, and decision panels](/research/evaluation-dashboard.png)

## Two pointers: `latest` vs `approved`

The trainer exports every new candidate to:

```text
models/latest.onnx
models/latest.meta.json
models/versions/{N}.onnx
```

Workers do **not** automatically follow `latest`. Self-play follows:

```text
models/approved.onnx
models/approved.meta.json
```

That split matters:

- `latest`: newest exported candidate from the trainer
- `approved`: incumbent model allowed to generate self-play

If a candidate fails evaluation, `latest` advances but `approved` does not.

## Core loop

For each new candidate version, the service does this:

1. Discover the current `approved` version and the newest candidate snapshot.
2. Run paired-color gate mini-matches: `candidate vs approved`.
3. Run paired-color benchmark mini-matches: `candidate vs Heuristic / Minimax-2 / Minimax-3 / Minimax-4`.
4. Persist every game as an immutable record with metadata including `eval_type`, `candidate`, `opponent`, `pair_id`, `pair_game_index`, `opening_seed`, and `opening_plies`.
5. Rebuild version-scoped summaries from the append-only record log.
6. Promote only if the gate passes and the benchmark regression checks pass.

Artifacts live under:

```text
state/evals/v{N}/gate_summary.json
state/evals/v{N}/benchmark_summary.json
state/evals/v{N}/decision.json
state/evals/v{N}/games/{ts}_{rand}.json
```

There is no global league output. Each candidate version carries its own evidence bundle.

## Paired-color opening diversification

Each mini-match is a **two-game pair**:

1. candidate as White, opponent as Black
2. opponent as White, candidate as Black

Both games reuse the same opening prefix via a shared `opening_seed`, then swap colors.

That matters because evaluation should measure the model, not one repeated deterministic opening or an asymmetric first-mover disadvantage. This diversification is for **measurement**, not for training data generation.

## Gate evidence

The direct gate is `candidate vs approved`. Each completed pair is reduced to one of five pentanomial buckets from the candidate's perspective:

| Pair result | Candidate score over 2 games | Bucket |
|-------------|-------------------------------|--------|
| loss + loss | `0.0` | `0.0` |
| loss + draw | `0.5` | `0.5` |
| draw + draw or win + loss | `1.0` | `1.0` |
| win + draw | `1.5` | `1.5` |
| win + win | `2.0` | `2.0` |

The shipping gate uses a paired pentanomial SPRT:

- minimum sample: `20` games
- hard cap: `100` games
- score hypotheses: `0.50` vs `0.55`
- error rates: `alpha = beta = 0.05`

The service also records a simple confidence interval view alongside the SPRT state so the dashboard can show both the formal decision and an easier-to-scan score band.

## Fixed-anchor benchmark suite

The benchmark suite uses the same paired-color format against fixed opponents:

- `Heuristic`
- `Minimax-2`
- `Minimax-3`
- `Minimax-4`

For the current approved model, the service first backfills a benchmark baseline so future candidates have a reference point. For later candidates, each anchor check compares the candidate's score against the approved model's anchor score minus the configured regression tolerance.

Each summary includes:

- `wins`, `losses`, `draws`
- paired-color bucket counts
- color split
- confidence / SPRT status
- per-opponent results
- move-time stats
- termination mix

## Promotion rule

The production rule is explicit:

- the candidate must pass the direct gate against the approved model
- the candidate must not regress beyond tolerance against the anchor suite

If either condition fails, the candidate is rejected and `approved` stays unchanged.

## Reading artifacts

`gate_summary.json`, `benchmark_summary.json`, and `decision.json` are version-scoped and auditable. They are rebuilt from immutable game records, so a restart or a second replica does not require trusting local in-memory state.

Per-game records include:

- `eval_type`
- `candidate`
- `candidate_version`
- `approved_version`
- `opponent`
- `pair_id`
- `pair_game_index`
- `opening_seed`
- `opening_plies`
- `pair_buckets`

The dashboard reads those summaries directly so you can answer:

- which candidate is under review
- whether gate evidence is still pending, rejected, or promoted
- which anchor caused a regression failure
- whether weird artifacts are coming from color split, termination mix, or time usage

## Local replay tool

To replay a gate locally with verbose per-move logging:

```bash
uv run python -m training.evaluate_gate \
  --approved 1 \
  --candidate 2 \
  --simulations 800 \
  --max-pairs 50 \
  --stop-when-decided
```

To keep the pair summaries but suppress per-move ticks:

```bash
uv run python -m training.evaluate_gate \
  --approved 1 \
  --candidate 2 \
  --simulations 800 \
  --max-pairs 50 \
  --stop-when-decided \
  --quiet-moves
```

The script prints:

- the direct gate summary
- the paired pentanomial SPRT view
- cumulative pair-bucket counts after every mini-match

That makes it useful for tuning thresholds or debugging a surprising promotion decision before changing the live evaluation service.
