The training pipeline lives in `training/` and implements an asynchronous, distributed AlphaZero loop. All shared state lives in S3 — workers and the trainer coordinate through file polling, with no direct communication.

You'd run this pipeline if you want to train a stronger network from scratch (or resume training an existing one). The end product is an ONNX model file that the engine can load via the [Python](/docs/usage/python) or [TypeScript](/docs/usage/typescript) bindings. To run it locally end-to-end, see [Deployment](/docs/research/deployment) — `make docker-up` is the shortest path.

## Components

### Workers (`training/worker.py`)

Self-play workers run a continuous loop:

1. Poll S3 for the latest model version
2. Play `worker_batch_size` games using MCTS + the current model
3. Flush the positions, policies, and outcomes as `.npz` files to S3
4. Repeat

Workers can run anywhere — local machines, cloud VMs, Docker containers, Kubernetes pods. They only need S3 credentials and enough CPU for MCTS search.

### Trainer (`training/trainer_loop.py`)

A single trainer process:

1. Downloads recent `.npz` files from S3 (KataGo-style sublinear replay window)
2. Trains the network for `steps_per_cycle` steps, sampling uniformly from the buffer
3. Exports the updated model to ONNX
4. Uploads the new model version to S3
5. Repeats

The trainer periodically re-scans S3 (every `reload_interval` steps) to pick up fresh data within a cycle.

### Elo service (`training/elo_service.py`)

A continuous rating service that:

1. Rates every model version (and a fixed set of heuristic / minimax baselines) with **OpenSkill** — a Bayesian Plackett-Luce / Weng-Lin model rescaled to look Elo-like (μ=25 → 1500). The "conservative rating" used for ranking is `μ − 2σ`.
2. Picks pairings using `predict_draw` over the current rating distribution: the top 5% of pairs by draw probability are sampled uniformly, so games concentrate where the result is genuinely uncertain instead of on hopeless mismatches.
3. Runs **placement matches** for new model versions — they exclusively play baselines until σ has shrunk enough for them to be matched against other models.
4. Scales horizontally. Each replica plays one game at a time and writes the result as an immutable per-game object under `state/elo_games/{ts}_{rand}.json`. A derived `state/elo.json` projection is rebuilt from the game log (last-writer-wins, idempotent), so multiple replicas never race on a shared state file.
5. LRU-caches loaded ONNX sessions (default 6 slots) to bound memory.

Position count is encoded in each `.npz` filename (`_n{count}`) so the trainer can count total positions via S3 LIST without downloading or opening files.

## S3 bucket layout

```
models/
  latest.onnx              # current model for inference
  latest.meta.json         # {"version": N, "timestamp": "...", "positions_at_promote": M}
  checkpoint.pt            # PyTorch training checkpoint
  versions/{N}.onnx        # immutable version snapshots

data/
  selfplay/v{N}/{ts}_{rand}_n{count}.npz
  selfplay_traces/v{N}/{game_id}.json
  imitation/{ts}_{rand}_n{count}.npz

state/
  elo.json                 # derived Elo projection (rebuilt from elo_games/)
  elo_games/{ts}_{rand}.json  # one immutable object per played game
  trainer_metrics.json     # latest trainer telemetry for the dashboard

heartbeats/
  {hostname}.json           # worker liveness + stats

benchmarks/
  results/{version}.json    # benchmark snapshots for the dashboard
```

## Training rate control

The trainer uses a **KataGo-style token bucket** to throttle training relative to data production. The ratio is controlled by:

$$
R = \frac{\text{training steps}}{\text{new data points}} \leq R_{\max} = 4.0
$$

If the trainer is outpacing data production, it sleeps rather than overfitting on stale data.

## Key hyperparameters

### Self-play

| Parameter | Default | Effect |
|-----------|---------|--------|
| `num_simulations` | 800 | MCTS sims per move — strength vs throughput |
| `temperature_threshold` | 60 | Move after which $\tau$ drops |
| `temperature_high` / `low` | 1.0 / 0.35 | Exploration vs clean endgame signals (Lc0-style floor) |
| `dirichlet_alpha` / `epsilon` | 0.3 / 0.25 | Root noise for exploration |

### Training

| Parameter | Default | Effect |
|-----------|---------|--------|
| `batch_size` | 256 | Training batch size |
| `learning_rate` | 5e-4 | SGD learning rate after warmup |
| `l2_regularization` | $3 \times 10^{-5}$ | Weight decay |
| `window_c` / `window_alpha` / `window_beta` | 25,000 / 0.75 / 0.4 | Sublinear replay-window parameters |
| `steps_per_cycle` | 3,000 | Steps between model promotions |

## Key dynamics

**Worker throughput vs training speed**: Workers produce positions proportional to `num_simulations` and available CPUs. The trainer consumes them based on `steps_per_cycle` and GPU speed. The token bucket keeps them balanced.

**Replay window vs cycle length**: The replay window grows sublinearly with total self-play volume, so early versions train mostly on fresh data while later versions retain more historical diversity without a hard linear-growth cost.

**Model version turnover**: Every `steps_per_cycle` steps, a new version is promoted. Workers poll for updates after each batch of games. The lag between promotion and adoption is a few minutes at most.
