The board is the core data structure of the engine, defined in `engine/src/board.rs`.

## Axial coordinates

Every cell is addressed by an **axial coordinate** `(q, r)`. A coordinate is valid when:

$$
\max(|q|, |r|, |q + r|) \leq 5
$$

This gives exactly 91 valid cells. The center of the board is `(0, 0)`.

<CoordinateBoard />

The third axis $s = -(q + r)$ is implicit — any two of $(q, r, s)$ determine the third. The validity constraint is equivalent to requiring all three axes to be in $[-5, 5]$.

## Cell indexing

For fast array access, each `(q, r)` coordinate maps to an integer index 0–90 via precomputed const lookup tables. This avoids hash maps in the hot path.

```rust
// Coordinate → index (fast, const table)
let idx = coord_to_index(HexCoord::new(0, 0)); // center = some index

// Index → coordinate (reverse lookup)
let coord = index_to_coord(idx);
```

## Board state

The `Board` struct stores:

- **`cells: [Option<Piece>; 91]`** — one slot per valid hex cell, `None` if empty
- **`en_passant: Option<HexCoord>`** — en passant target square, if any
- **`zobrist: u64`** — incrementally updated Zobrist hash for transposition detection

Each `Piece` has a `kind` (King, Queen, Rook, Bishop, Knight, Pawn) and a `color` (White, Black).

## Zobrist hashing

The board maintains a **Zobrist hash** that is incrementally updated on every move. This is used for:

- **Threefold repetition detection** — the `GameState` tracks a history of Zobrist hashes
- **Transposition table** in MCTS — caching neural network evaluations for positions that arise via different move orders

The hash incorporates piece positions, side to move, and en passant state.

## Tensor encoding

For neural network input, the board maps into an **11x11 grid** using `col = q + 5, row = r + 5`. Invalid cells (outside the hex) are zero-padded. The current encoder is **side-to-move framed**: when Black is to move, the board is centrally inverted and colors are swapped so the network always sees "my pieces" moving toward "my" promotion edge. The full tensor shape is:

$$
(22, 11, 11) = \text{channels} \times \text{rows} \times \text{cols}
$$

| Channels | Content |
|----------|---------|
| 0–5 | Side-to-move piece planes (pawn, knight, bishop, rook, queen, king) |
| 6–11 | Opponent piece planes |
| 12 | Absolute side to move |
| 13 | Fullmove number |
| 14 | Halfmove clock |
| 15 | En passant target |
| 16 | Repetition count |
| 17 | Validity mask (1 for valid hex cells, 0 for padding) |
| 18 | In-check flag |
| 19 | Last-move from square |
| 20 | Last-move to square |
| 21 | Plies since pawn move (derived from the halfmove clock) |

The policy vector is defined in the same frame. For training targets, use the STM-aware helpers (`game.policy_index(...)` in Python or `stm_policy_index` in Rust) so Black-to-move targets land in the same slot the network emits.

See the [Python bindings](/docs/usage/python#board-encoding-for-training) for usage.
